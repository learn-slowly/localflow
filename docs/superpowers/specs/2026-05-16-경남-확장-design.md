# 경남 전역 확장 설계

작성일: 2026-05-16
대상 브랜치: main

## 1. 배경

LocalFlow는 진주시 단일 도시 분석 도구로 시작했지만, `src/config/cities/`에 경남 22개 시군구 설정이 이미 갖춰져 있고 `public/data/nationwide-*.json` 캐시도 마련되어 있다. 이번 작업은 **첫 진입 화면을 경남 전역 지도로 바꾸고, 진주 분석 화면은 `/jinju` 라우트로 옮긴다**. 다른 21개 시군구도 인구·선거구 레이어 수준에서 동일한 화면 구조를 제공한다.

## 2. 현재 상태 (작업 전)

- `/` = 진주 선거구 브리핑 카드 리스트 (`src/app/page.tsx`)
- `/map` = 진주 카카오맵 전체 화면 (`src/app/map/page.tsx`, 미커밋)
- `/district/[id]` = 진주 선거구 상세 (`src/app/district/[id]/page.tsx`, 미커밋)
- `MapContainer` (1152라인) 안에서 진주 데이터 직접 import
- 도시 설정은 22개 시군구 + 진주 + 파주 모두 존재, `GYEONGNAM_VIEW` 상수도 있음

## 3. 목표 라우트 / 화면

```
/                          경남 전역 지도 (기본 진입)
  · 22개 시군구 폴리곤 (인구 코로플레스)
  · 선거구 레이어 — 기초/도/국/시장 탭 전환 (진주만 데이터, 다른 곳은 "준비 중" 배지)
  · 시군구 클릭 → /[city] 라우트로 이동

/[city]                    도시별 지도 (사이드 패널에 브리핑)
  · /jinju = 전체 레이어 활성 (인구·선거구·교통·경로당·상권·핀·사진 등 현재 진주가 가진 모든 레이어)
  · 그 외 = 인구 레이어만 활성. 선거구·교통·경로당·상권 등은 토글 disabled + "준비 중" 배지

/[city]/district/[id]      선거구 상세 (기존 /district/[id]를 도시 하위로)

/map                       제거 (지도가 / 와 /[city]의 메인이 됨)
/district/[id]             제거 (/[city]/district/[id]로 이동)
```

경로 가드: `[city]` 동적 세그먼트는 `src/config/cities/index.ts`의 `cities` 키 화이트리스트로 검증. 미등록 도시는 `notFound()`.

## 4. 데이터 흐름

### 4.1 빌드타임 가공

새 스크립트 `scripts/build-gyeongnam-summary.ts`:
- 입력
  - `public/data/nationwide-population.json` (10자리 법정동 코드 키, 1.3MB)
  - `public/data/nationwide-boundary.json` (HangJeongDong GeoJSON, properties에 `sgg` 시군구코드·`sido` 시도코드 포함)
  - `src/config/cities/index.ts`의 22개 경남 `CityConfig`
- 출력
  - `src/data/gyeongnam-cities.json` — 22개 시군구별 인구 요약
    ```ts
    {
      [cityCode: string]: {
        name: string;
        totalPopulation: number;
        households: number;
        male: number;
        female: number;
        center: [number, number];
      }
    }
    ```
  - `public/data/gyeongnam-boundary.json` — 22개 시군구 단위 단순화된 폴리곤 GeoJSON. 전국 GeoJSON에서 `sido === "48"`(경남)을 필터하고 `sgg` 기준으로 동을 합쳐(dissolve) 시군구 경계 생성.

> dissolve는 단순화를 위해 turf.js 또는 동등 라이브러리 사용. 결과 파일은 1MB 이내 목표.

### 4.2 런타임

| 라우트 | 사용 데이터 |
|---|---|
| `/` | `src/data/gyeongnam-cities.json`, `public/data/gyeongnam-boundary.json`, `jinju-elections.json`/`jinju-local-elections.json` (진주 선거구만) |
| `/jinju` | 기존 진주 데이터 일체 (`jinju-*.json`) |
| `/[other-city]` | `gyeongnam-cities.json[cityCode]` 만, 그 외 레이어는 빈 상태 |

## 5. 컴포넌트 분할

### 5.1 새 컴포넌트

- `src/components/Map/GyeongnamMap.tsx` (NEW) — 경남 전역용. 22개 시군구 폴리곤, 인구 코로플레스, 선거 유형 탭(기초/도/국/시장), 시군구 클릭 시 `router.push("/" + cityKey)`. 사이드 패널 없음(상단 탭만). 선거구 레이어는 진주 영역에서만 폴리곤이 그려지고 다른 시군구는 경계만 표시.
- `src/app/[city]/page.tsx` (NEW) — 도시별 메인. `MapContainer`에 도시 설정과 데이터 가용성 plain을 전달.
- `src/app/[city]/district/[id]/page.tsx` — 기존 `src/app/district/[id]/page.tsx`를 이동.

### 5.2 기존 `MapContainer` 일반화

현재 `MapContainer`는 진주 JSON을 직접 import한다(`jinjuBoundary`, `jinjuLegalBoundary`, `jinjuBusStops`, `jinjuTransitUsage`, `jinjuDistricts`). 1152라인 단일 파일이라 한 번에 다 정리하지 않고, **얇은 prop 인터페이스**만 추가해 진주 페이지가 같은 데이터를 명시적으로 주입하게 바꾼다.

```ts
type MapContainerProps = {
  city: CityConfig;
  availability: {
    boundary: boolean;
    legalBoundary: boolean;
    busStops: boolean;
    transitUsage: boolean;
    districts: boolean;
    seniorCenters: boolean;
    commerce: boolean;
  };
  data?: {
    boundary?: any;
    legalBoundary?: any;
    busStops?: any;
    transitUsage?: any;
    districts?: any;
    seniorCenters?: any;
    commerce?: any;
  };
};
```

`/jinju`에서는 모든 가용성 `true` + 진주 JSON을 주입. 다른 도시는 가용성 모두 `false`. 가용성이 `false`인 레이어는 토글 UI에 "준비 중" 배지로 표시하고 마운트하지 않는다.

> MapContainer 내부 리팩토링(컴포넌트 분할)은 이번 작업 범위 밖. prop 인터페이스만 도입한다.

### 5.3 사이드 패널 / 브리핑

`/[city]`의 사이드 패널 (PopulationPanel, ElectionPanel 등)은 기존 진주 브리핑 데이터를 그대로 사용. 다른 도시는 인구 패널만 표시하고 선거 패널은 "데이터 준비 중".

기존 `/`(브리핑 카드)는 사라지지만, 같은 컨텐츠를 사이드 패널에 카드 형태로 띄우는 방향. `src/lib/district-recommender.ts`의 `buildAllBriefings`는 그대로 재사용.

## 6. 빈 상태 / 도시별 데이터 가용성

- 인구·선거구 외 레이어는 진주만 가용. 다른 도시 진입 시:
  - 토글 버튼은 보이되 disabled + "준비 중" 배지
  - 사이드 패널의 해당 섹션은 스켈레톤 또는 안내 문구
- 디자인 톤: 빈 섹션을 가리지 말고 "이 도시는 인구 데이터만 표시합니다" 식 안내를 명시.

## 7. 미커밋 변경분 처리

작업 직전에 다음을 우선 커밋·이동한다.

| 파일 | 처리 |
|---|---|
| `src/app/page.tsx` | 기존 진주 브리핑을 `/jinju/page.tsx`로 옮긴 후, `/`를 경남 지도로 교체 |
| `src/app/globals.css` | 그대로 |
| `src/app/map/page.tsx` | 삭제 (지도는 `/`와 `/[city]`가 흡수) |
| `src/app/district/[id]/page.tsx` | `src/app/[city]/district/[id]/page.tsx`로 이동, 데이터 import는 도시 파라미터에서 분기 |
| `src/lib/district-recommender.ts` | 그대로 유지 |

## 8. 작업 단계 (대략)

1. **빌드 스크립트** 작성·실행 → `gyeongnam-cities.json`, `gyeongnam-boundary.json` 생성
2. **`MapContainer` prop 인터페이스 추가** — 직접 import → props 주입 경로로 변경 (진주 페이지에서 기존 JSON 명시 주입)
3. **`/[city]` 동적 라우트** 신설, 기존 진주 브리핑 컴포넌트 이동·일반화
4. **`/jinju` 동작 확인** — 진주 페이지가 새 라우트에서 기존과 동일하게 동작
5. **`GyeongnamMap` 신규 컴포넌트** 작성, `/`를 경남 지도로 교체
6. **`/[city]/district/[id]`로 이동** — 기존 `/district/[id]` 삭제
7. **`/map` 삭제**
8. 다른 시군구 페이지 (인구만)에서 빈 상태 UX 확인

## 9. 비범위

- MapContainer 1152라인 내부 리팩토링·컴포넌트 분할
- 다른 시군구의 교통·경로당·상권·선거구 데이터 수집
- DB(Supabase) 도입
- 다른 광역시도(경기·서울 등) 확장 (`paju.ts`는 그대로 두되 라우팅 불노출)
- 카카오맵 외 지도 라이브러리 변경

## 10. 후속 후보

- 다른 광역시도용 `RegionView` 일반화 (현재는 경남 전용)
- 시군구 단위 인구 외 통계(고령화율, 면적 등) 카드 추가
- 다른 도시도 교통 OD 데이터 수집 시 같은 도시 페이지에 자동 활성
